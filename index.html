Go Further

  • This is a post mortem of the game project.
  • There are many good ideas that have not be implemented, limited by time and energy. For example, it comes to me that setting bricks with special effects might make the game more interesting, like a brick that explodes after hit, or a rainbow brick that appears randomly among the bricks and values extra high points. The introduction of various special effects take very few efforts but will make the game stand out, and more attractive. Unfortunately I focus more on the game engine design and the interactions, so those works remain to be done in the future.
  • As for the engine itself, the physics of the game could be really simple. It just includes basic collision and reflection. But the game with such a physical model is really dull as the movement of the ball is completely predictable, and independent of user operations. In order to make a difference, I introduce a refraction angle after every bounce. But this would in the opposite makes the velocity direction unbalanced, i.e. the speed of x or y axis gets too big. By defining different behavior after ball collision with the bricks or the paddle I temporarily solve the problem, but it hits me that to design a good physic model for object movement is still a problem.
  • Here comes to the interface and interaction of the game. It provides basically 3 instructions of quitting, skipping and restarting. But once I would like to introduce more instructions, the menu has to be multi-leveled, and the complexity rapidly increases ( I don't like the interface fully crowed with prompts.) Moreover, there should be some animation effects if I would like to make the game more user-friendly. But this is another big challenge remains challenging.
  • There occur many problems when developing the game. For instances, I kept on OOP and implemented different classes individually. But as for the single main class, it got increasingly difficult to add new features to the main class. When I tried to integrate the count down function into the fps-capped loop instead of using SDL_Delay(1000), there were too many variables involved and it was difficult to do that. Besides, the control logic became a mess after adding various functions. It took me much time to refactor that. I found it important to have a thorough diagram of each class before code implementation.
  • Last but not least, the documentation and publishment are much more time-costing than I thought before. It took about 6 hours to do, and the outcoming is even not so satisfying. I spent most of my time on the coding part, and in the next project I would like to rearrange my time schedule.
